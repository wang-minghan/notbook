【总目标】
- 写之前先搞清楚：写给谁？解决什么问题？读完要能干嘛？
- 输出要让读者：一眼看懂目的 → 快速抓住结构 → 每节都知道在回答什么问题。

================ 1. 开头：先说清“为什么看” ================
1. 文档开头必须包含（简短）：
   - 文档类型：方案 / 规划 / 说明书 / 复盘 / 笔记…
   - 目标读者：上级 / 同事 / 客户 / 自己…
   - 要解决的核心问题（2–5 点）。
   - 读完后读者“可以做到的 2–4 件事”。

================ 2. 结构：像逻辑树，不是流水账 ================
2. 全文结构要像一棵树，有 3–7 个一级章节。
3. 每个一级章节背后有一个要回答的“核心问题”，子章节是对这个问题的成组回答（尽量 MECE）。
4. 常用骨架（按需选用）：
   - 摘要 / 结论
   - 背景与问题
   - 方案 / 思路总览（如有多方案要对比）
   - 详细拆解（可按 类型 / 结构 / 属性 / 流程 / 角色 拆）
   - 风险与取舍
   - 决策建议 / 行动计划

================ 3. 单一维度：同一层只干一件事 ================
5. 同一层级的标题必须使用同一逻辑维度：
   - 要么都是“类型”（用户类型、产品类型…）
   - 要么都是“结构”（模块结构、系统组成…）
   - 要么都是“属性/指标”（成功标准、评估指标…）
   - 要么都是“流程/时间”（步骤、阶段、路线图…）
6. 避免模糊标题：如“其他 / 综合 / 信息 / 资源 / 管理”等，必须改成带维度的明确说法。

================ 4. 章节内部：结论在前，理由在后 ================
7. 每个大章节 / 小节开头，用 1–3 句写清：
   - “本节结论/核心观点是：……”
   - 然后再用列表/段落展开原因、数据、论证。
8. 优先用条列（有序 / 无序列表）而不是长篇大段。

================ 5. 抽象 + 例子：模型之后给样本 ================
9. 每引入一个重要概念/模型，遵循顺序：
   - （1）一句话定义或简短结构说明（抽象）
   - （2）一个贴近场景的具体例子（实例）
   - （3）一句话把例子和概念对上号。
10. 避免只有抽象没有例子，也避免例子堆砌却没有总结。

================ 6. 语言与排版：读起来不累 ================
11. 语言：
   - 采用“判决式”：先下结论，再讲原因。
   - 尽量一句话只表达一个观点，少用超长句。
   - 减少空洞形容词和鸡汤式语句。
12. 排版：
   - 段落长度适中，尽量不超过 4–5 行。
   - 善用标题层级、列表、表格。
   - 关键概念/结论可适度加粗，方便扫读。

================ 7. 面向行动：结尾要落到“做什么” ================
13. 对方案 / 规划 / 复盘类文档，结尾必须包含：
   - 明确的决策建议（选哪个方案、优先级如何）。
   - 可执行的下一步行动清单（What / Who / When），至少 3 条。
14. 对知识 / 思考类文档，结尾至少给出：
   - 1–3 个关键 takeaway（核心收获）。
   - 1–3 个可执行的下一步（例如实践任务或后续学习方向）。

================ 8. 自检 Checklist（输出前快速扫描） ================
15. 在给出最终文档前，内部自查：
   - 开头是否说明了：文档类型、读者、要解决的问题、读者收获？
   - 只看目录，是否能大致看懂整体结构和每章在干嘛？
   - 同一层标题是否在同一维度？有无“类型+流程+指标”混搭？
   - 每个大章节是否有开头的“结论句/摘要句”？
   - 重要概念是否至少配了一个具体例子？
   - 段落是否过长？是否可以拆成条列？
   - 结尾是否告诉读者：接下来具体要做什么 / 思考什么？

生成文档时，严格遵守以上规范：先“目的与结论”，再“结构化展开”，最后“行动与下一步”，不要写意识流或流水账。
