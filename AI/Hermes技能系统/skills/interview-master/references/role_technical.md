# 技术/研发岗面试专项指南

---

## 一、面试官在考什么（考察维度）

### 技术面试核心维度

| 维度 | 权重 | 面试官心里的标准 |
|------|------|----------------|
| 编码能力 | 25% | 能不能写出正确、干净、有效率的代码 |
| 系统设计 | 20% | 面对复杂系统能否做出合理的架构决策（社招权重更高） |
| 计算机基础 | 15% | 数据结构、算法、操作系统、网络等基础是否扎实 |
| 工程素养 | 15% | 代码质量、测试意识、性能意识、协作规范 |
| 问题解决 | 15% | 遇到未知问题时的分析和拆解能力 |
| 沟通与协作 | 10% | 能否清晰表达技术方案，与非技术角色有效沟通 |

### 社招额外维度

| 维度 | 考察内容 |
|------|---------|
| 技术深度 | 在某个领域是否有深入研究和独到见解 |
| 技术选型能力 | 能否在多个方案中做出合理的trade-off |
| 架构思维 | 是否能从系统全局视角思考问题 |
| 技术影响力 | 是否在团队中推动过技术改进或最佳实践 |

## 二、高概率面试问题

### 行为/软实力题（技术面也会问）

- 说一个你解决过的最难的技术问题？你的思路是什么？
- 你和团队成员在技术方案上产生分歧，你怎么处理的？
- 说一个你主导的技术改进/优化项目，效果怎么样？
- 你怎么保持技术成长？最近在学什么？
- 你觉得代码质量和开发速度之间怎么平衡？

### 技术基础题（校招重点）

**数据结构与算法：**
- 链表反转、二叉树遍历、动态规划基础题
- 排序算法的时间/空间复杂度对比
- 什么场景用HashMap、什么场景用TreeMap？

**操作系统：**
- 进程和线程的区别？什么时候用多线程、什么时候用多进程？
- 死锁的四个必要条件？如何预防？
- 虚拟内存的工作原理？

**网络：**
- TCP和UDP的区别？什么场景用UDP？
- HTTPS的握手过程？
- 从输入URL到页面展示，中间发生了什么？

**数据库：**
- 索引的原理？什么时候该建索引、什么时候不该？
- 事务的ACID特性？隔离级别有哪些？
- SQL优化有哪些常见方法？

### 系统设计题（社招重点）

- 设计一个短链服务
- 设计一个消息队列
- 设计一个朋友圈/Feed流系统
- 设计一个秒杀系统
- 设计一个分布式缓存
- 设计一个日志收集和分析系统

### 项目深挖题（社招核心）

面试官会挑你简历上写的项目，不断追问细节：
- 这个项目的整体架构是什么？你负责哪一部分？
- 为什么选择这个技术方案？有没有考虑过别的方案？
- 遇到的最大技术挑战是什么？你怎么解决的？
- 上线后有没有出过问题？怎么排查的？
- 这个系统的QPS是多少？做了哪些性能优化？
- 如果重新来过，你会怎么改进？

**应对策略：** 简历上写的每个项目，都要准备一个"深挖包"：
```
项目背景：为什么做这个？解决什么业务问题？
技术选型：用了什么技术？为什么选它？trade-off是什么？
核心设计：架构图能画出来，关键模块能讲清楚
挑战与解决：至少准备一个技术难点和你的解决方案
数据与结果：性能指标、业务指标、改进前后对比
反思与改进：如果重来会怎么做？学到了什么？
```

## 三、实操考核类型

### 1. 算法/编程题（校招必考，社招看公司）

**校招典型流程：**
- 笔试：2-3道算法题，90-120分钟，在线OJ
- 面试中：1-2道中等难度题，手写或IDE

**准备策略：**
```
基础（必须掌握）：
  - 数组/字符串操作、链表、栈/队列、哈希表
  - 二叉树遍历（前中后序、层序）
  - 二分查找、双指针、滑动窗口
  - 基础排序（快排、归并）
  - BFS/DFS

进阶（中大厂校招需要）：
  - 动态规划（背包、子序列、区间DP）
  - 图算法（最短路径、拓扑排序）
  - 单调栈/单调队列
  - 前缀和/差分数组
  - 并查集
```

### 2. 系统设计（社招3年以上几乎必考）

详细框架见 `case_study_guide.md` 系统设计部分。

**准备策略：**
```
必备知识模块：
  - 负载均衡：策略选型（轮询/加权/一致性哈希）
  - 缓存：Redis/Memcached，缓存穿透/击穿/雪崩
  - 消息队列：Kafka/RabbitMQ，保证顺序/不丢/不重
  - 数据库：分库分表、读写分离、主从同步
  - 分布式：CAP/BASE理论、分布式锁、分布式事务

刷题路径：
  经典系统设计题刷10-15道 → 总结通用模式 → 针对目标公司业务做专项准备
```

### 3. Code Review / 代码评审

**常见形式：** 给你一段200-500行的代码，让你找出问题并给改进建议。

### 4. Take-home Assignment

**常见形式：** 给你一个小项目，3-7天完成，然后面试时展示和答辩。

**应对策略：**
```
- 代码质量 > 功能数量：做好核心功能比做多做全重要
- 必须有README：说明如何运行、设计思路、技术选型理由
- 必须有测试：至少有单元测试，覆盖核心逻辑
- 不要过度设计：适当的设计就好，不需要搞微服务架构
- 提前完成：预留1-2天做code review和优化
```

## 四、简历标准

### 技术简历怎么写

**项目经历格式：**
```
[项目名称] | [技术栈] | [时间]
背景：用一句话说清楚这个项目解决什么问题
职责：你在团队中的角色和核心贡献
技术方案：关键技术决策（不要罗列所有技术栈）
结果：性能指标或业务指标的改进
```

**示例对比：**
```
❌ 弱：使用Spring Boot + MySQL + Redis开发了一个电商后台管理系统，
       实现了商品管理、订单管理、用户管理等功能

✅ 强：主导电商平台订单系统重构，将单体架构拆分为订单、库存、支付三个微服务，
       引入Redis缓存热点商品库存（命中率98%），设计基于消息队列的最终一致性方案
       解决分布式事务问题，系统QPS从800提升至5000+，99分位响应时间从2s降至200ms
```

**技术简历致命错误：**
- 罗列技术名词但不说用在什么场景
- 只写"参与开发"，看不出你的贡献
- 没有任何可量化的技术指标
- 把所有学过的技术都堆在技能列表里

## 五、校招 vs 社招差异

| 维度 | 校招 | 社招（3年以下） | 社招（3年以上） |
|------|------|--------------|--------------|
| 算法题 | 必考，中高难度 | 看公司，中等难度 | 部分公司考，可能较简单 |
| 系统设计 | 基本不考 | 简单场景 | 核心考核项 |
| 项目深挖 | 实习/课程项目 | 工作项目（深度适中） | 工作项目（深挖到底） |
| 基础知识 | 重点考察 | 结合项目考 | 按需考察 |
| 面试轮数 | 3-4轮 | 3-4轮 | 4-5轮（含CTO面） |

### 面试前的技术准备清单

```
□ 目标公司的技术栈了解清楚了吗？（可以从招聘JD、技术博客、开源项目了解）
□ 简历上每个项目的"深挖包"准备好了吗？
□ 算法题刷够了吗？（校招建议至少200题，社招100题）
□ 系统设计的常见题型准备了吗？（社招必备）
□ 最近有没有关注目标公司所在领域的技术趋势？
□ 能否用非技术语言解释你做的项目？（终面可能会面对非技术面试官）
```
