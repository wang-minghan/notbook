# 知识库治理规范

> 这页只管一件事：让自动任务别把“目录结构”当“语义结构”，也别把时间页塞进全局目录。

## 页面分类
- **hub**：总入口、主题地图、索引、规范、日志
- **entity / concept / comparison / query / workflow**：长期可复用的知识页
- **time**：日计划、周计划、阶段计划、会议纪要、临时执行页
- **generated**：自动任务产物、增量摘要、lint 结果

## 范围规则
- **global**：只给长期稳定、会反复被引用的页
- **section**：只在某个子目录内成立的入口页
- **local**：时间页、临时页、一次性产物

## 索引规则
1. `index.md` 只收 **global** 页。
2. `日计划 / 周计划 / 临时计划` 不进全局 index，只进各自子目录的 section index。
3. 父目录和子目录只是存储层级，不自动代表语义层级。
4. 只有当某个目录真的形成稳定主题时，才允许创建该目录的 hub / index。
5. 时间页如果被长期复用，先升级成 workflow 或 query，再决定是否进总入口。
6. 每个稳定模块都应该有自己的入口页；如果没有，就先补入口页，再谈是否进全局索引。

## 增量压缩摘要
- 自动任务不要每次重写全库总结，只处理 **新增 / 修改 / 删除** 的文件。
- 每次只产出一条短摘要，写入 `知识库增量摘要.md`。
- 摘要只写“变了什么”和“为什么要变”，不要重复旧内容。
- 当摘要条目累计到一个窗口，就合并压缩；如果文件过长，滚到 `知识库增量摘要-YYYY-MM.md`。

## 需要重点防的风险
- 把周计划、日计划误加进总索引，导致 index 膨胀
- 把目录层级误当成知识层级，导致链接关系失真
- 没有增量摘要，导致自动任务每次都像第一次见这个库
- 过度总结，把 stable 页面也反复改写，制造版本漂移
- 只修链接不修类型，最后页面越连越乱

## 处理建议
- 先判定页面类型，再决定是否进全局 index
- 先找 hub，再找叶子页，别反过来
- 对时间页只做局部连接，不做全局编目
- 对自动任务产物统一写进 `generated` 层
- 每次结构性改动都回写 `log.md`
