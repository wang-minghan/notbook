【项目知识固化规则：小型代码知识图谱】

1. 你是本项目的“长期合作者”，要在多轮对话中逐步建立并维护一个小型代码知识图谱，而不是每次当新手。

2. 图谱里至少包含这些实体：
   - Pipeline：数据/训练/预测流水线名。
   - Dataset / Feature：数据集名、重要特征列名。
   - Model：模型类/构建函数名。
   - Config：配置文件名或关键配置键。
   - Test：测试文件/用例名。
   - Bug / Lesson：重要问题及教训的简短描述。

3. 图谱里至少包含这些关系：
   - pipeline 使用哪些数据/特征：uses_dataset(pipeline, dataset)、uses_feature(pipeline, feature)。
   - pipeline 训练/调用哪些模型：trains_model(pipeline, model)、applies_model(pipeline, model)。
   - 模型由哪些配置驱动：configured_by(model, config)。
   - 哪个测试覆盖哪些函数/模块：tests_function(test, function/module)。
   - 哪个测试守护/复现过哪个 bug：guards_bug(test, bug)。

4. 每当你新了解到：
   - 一个新的 pipeline / 模型 / 配置 / 测试；
   - 或定位并修复了一个新的 bug / 教训，
   都应尝试用 MCP 或可用的持久化机制，把对应的实体和关系写入项目知识库（没有工具就用结构化文本总结，方便外部存）。

5. 在后续对话中，当用户提到已存在于图谱中的 pipeline / 模型 / 配置 / 测试 / bug 时：
   - 先查询或回忆已有节点和关系，再给建议或写代码；
   - 避免忽略已有约定和教训，尽量保持前后一致，并在有新理解时更新旧知识，而不是产生矛盾版本。
